# 开发推进计划

## 1. 计划说明

本计划面向当前仓库初始化阶段，按照 3 人团队的实际协作强度设计，优先目标是建立可持续迭代的基础骨架，而不是一次性堆叠所有业务功能。

建议采用“按月推进 + 每月明确产出 + 每阶段至少一条可演示链路”的方式推进。

## 2. 阶段划分

### 阶段 0：仓库初始化与契约收敛

时间建议：2026-04

目标：

- 完成 monorepo 结构与目录约束。
- 收敛共享类型、任务模型、风险等级模型。
- 明确前后端与三个引擎的边界。

预期产出：

- 根目录说明文档
- 架构文档
- API 契约初稿
- 前后端与引擎 README
- 第一版协作分工与命名规范

### 阶段 1：最小链路打通

时间建议：2026-05

目标：

- 打通“创建任务 -> 执行检测 -> 回传结果 -> 页面展示”的最小闭环。
- 后端完成任务中心雏形。
- 前端完成任务列表与结果详情基础页面。
- 选一个引擎优先跑通，建议先从 `skills-static` 或 `asset-scan` 开始。

预期产出：

- 后端最小 API
- 前端最小页面
- 一个引擎的模拟或真实执行链路
- 集成联调样例
- 第一版端到端演示

### 阶段 2：三类能力骨架成型

时间建议：2026-06

目标：

- `asset-scan`、`skills-static`、`sandbox` 三个引擎都具备可运行骨架。
- 平台后端支持按任务类型调度不同引擎。
- 统一结果存储与风险汇总视图。

预期产出：

- 三类任务类型可创建
- 三类结果模型可落库或归档
- 统一风险等级展示
- 样本数据与联调脚本
- 初版集成测试

### 阶段 3：结果汇总与平台化增强

时间建议：2026-07

目标：

- 增强平台化能力，包括筛选、检索、结果聚合、趋势统计。
- 强化沙箱告警流与事件追踪能力。
- 补充部署、日志、审计等非功能骨架。

预期产出：

- 风险汇总页面
- 资产详情页与分析报告页
- 沙箱告警看板
- 本地部署脚本初版
- 基础审计字段与操作日志方案

## 3. 月度推进框架

### 2026-04：打基础

重点事项：

- 定目录、定命名、定契约
- 完成 shared 核心模型
- 明确任务中心边界

建议责任分配：

- 成员 A：前端项目初始化与页面骨架
- 成员 B：后端模块骨架与 API 入口
- 成员 C：引擎目录、规则组织和样本准备

### 2026-05：跑通一条链路

重点事项：

- 最小任务流打通
- 首个引擎接入后端
- 页面展示最小结果

建议验收标准：

- 能创建任务
- 能看到执行状态变化
- 能查看至少一种结果详情

### 2026-06：扩三类能力

重点事项：

- 三个引擎都能接入平台
- 风险等级统一
- 样本驱动测试初步建立

建议验收标准：

- 三类任务均可创建
- 至少各有一份样例结果
- 后端可统一查询任务和结果

### 2026-07：平台化与可运维性增强

重点事项：

- 汇总视图与告警视图
- 部署脚本骨架
- 基础审计与日志链路

建议验收标准：

- 管理台具备可演示页面
- 本地可启动完整开发环境
- 关键模块有基础测试覆盖

## 4. 分阶段目标

### 4.1 工程目标

- 形成统一 monorepo 工作方式
- 建立共享契约和命名规范
- 支持多模块并行开发

### 4.2 业务目标

- 具备资产测绘、Skills 静态检测、沙箱监控三条核心能力骨架
- 平台可以统一承载任务与结果
- 风险数据可以被结构化展示与追踪

### 4.3 协作目标

- 每个成员都有清晰边界，同时不过度割裂
- 接口以文档和 `shared` 契约为准
- 联调优先围绕样本和最小链路展开

## 5. 每阶段预期产出汇总

| 阶段 | 核心目标 | 预期产出 |
| --- | --- | --- |
| 2026-04 | 仓库初始化 | 目录骨架、契约草案、架构文档 |
| 2026-05 | 最小链路 | 任务流、首个引擎接入、基础页面 |
| 2026-06 | 三引擎骨架 | 三类任务、统一结果结构、集成样例 |
| 2026-07 | 平台增强 | 汇总展示、告警看板、部署骨架 |

## 6. 风险与提醒

- 如果 `shared` 契约频繁变动，会直接影响前后端和引擎联调效率。
- 如果后端承担过多检测细节，后续会削弱引擎独立性。
- 如果没有尽早准备 `samples/` 样本，联调阶段会出现大量临时数据拼装成本。
- 对 3 人团队而言，第一阶段最重要的是接口稳定和边界清晰，而不是功能数量。
