## 一、阶段目标与背景

本阶段工作围绕“智能体资产测绘任务”展开，目标不是做一次性的演示脚本，而是在现有平台骨架中沉淀一条可长期演进的资产识别链路。项目整体采用“后端统一任务编排、引擎独立执行、共享契约统一输出”的结构，当前重点落在 `asset-scan` 引擎和后端 `task-center` 的联调闭环上。

在功能目标上，本阶段主要解决三个问题：

- 将资产测绘结果从文档和样例整理为工程化类型、规则和输出契约。
- 将 `asset-scan` 从单点探针扩展为“发现、端口、协议、探针、指纹、风险”的分层流程。
- 将 FOFA 公网候选、后端任务链路、naabu/nmap 端口与服务验证、HTTP `/api/tags` 补证整合为可复跑的评估基线。

## 三、资产测绘工程化结构

前期整理的资产测绘材料已经被落地为项目中的标准结果结构，核心字段包括：

- `target`：扫描目标及输入来源。
- `asset`：资产 IP、域名、来源和采集时间。
- `network`：开放端口、协议、服务和端口状态。
- `application`：HTTP endpoint、认证状态和接口探测结果。
- `fingerprints`：命中的智能体、LLM 服务或 Agent 框架指纹。
- `inferred_attributes`：推断出的服务、框架、模型来源和部署方式。
- `findings`：风险发现、风险等级、证据、原因和修复建议。

这些结构已经对应到 `shared/types/asset-scan.ts`、`shared/types/result.ts`、`engines/asset-scan/src/runtime/classification.service.ts` 以及后端任务结果回填逻辑中。这样设计的好处是，后端和前端看到的是平台统一结果，具体探测细节仍由引擎内部维护，避免平台层直接耦合某个引擎实现。

## 四、规则与引擎实现进展

当前 `asset-scan` 已经完成 v2 规则基线。指纹规则采用加权信号模型，支持 `open_port`、`api_path`、`json_key`、`http_body`、`ws_message` 等特征，并支持 `equals`、`contains`、`has_key`、`in` 等匹配算子。重点目标包括 `ollama`、`langflow`、`autogpt`、`openclaw-gateway`，同时预留了 `mcp-server`、`dify`、`anythingllm` 等后续扩展目标。

探针规则按照层级划分：

- L1：端口存活探测。
- L2：HTTP API endpoint 探测。
- L3：WebSocket 探测。

当前固化的重点端口包括：

| 等级 | 端口 |
| --- | --- |
| critical | 18789、18791、18793 |
| high | 11434、3000、8080、9090 |
| medium | 8000、5000、3001 |
| low | 80、443、8081 |

引擎主流程已经按六步实现，入口集中在 `engines/asset-scan/src/runtime/pipeline.ts`、`engines/asset-scan/src/runtime/run-task.ts` 和 `engines/asset-scan/src/bridge/scan-task.ts`。其中后端只负责创建任务、调度引擎和保存结果，引擎负责具体探测、指纹识别与风险输出。

## 五、FOFA 与分层扫描闭环

本阶段新增的关键闭环是：通过 FOFA 工具面向 Ollama 暴露面的查询模板筛选公网候选，再通过 task-scan 接入现有后端任务链路，形成 naabu 端口探测、nmap 服务验证、HTTP `/api/tags` 补证的分层扫描与回退流程，并进一步形成目标结果、正负样本、评估对比和执行基线。

具体流程如下：

1. 使用 FOFA 查询模板筛选候选目标，例如围绕 `app="Ollama" && is_domain=false` 获取公网候选。
2. 通过 dev 侧 task-scan 脚本把 FOFA 记录转换成平台现有 `asset_scan` 任务请求。
3. 后端通过 `POST /api/tasks` 创建任务，并通过统一结果接口读取 `result` 和 `risk-summary`。
4. 对候选目标执行 naabu 端口开放探测，naabu 只负责判断端口是否开放。
5. 仅对命中端口执行 nmap 服务验证，采集 `http-title`、`ssl-cert`、`http-enum` 等证据。
6. 对 Ollama 候选执行 HTTP `/api/tags` 补证，将可解析、含模型字段的响应作为强正样本依据。
7. 将结果分为强正样本、强负样本和运输失败样本，运输失败只保留证据，不进入正负样本库。

该流程保持了两个边界：一是公网扫描必须在授权范围与预算控制下执行；二是 FOFA 和端口工具属于 dev 侧执行与证据积累层，不新增平台公开路由，不破坏后端统一任务 API。

## 六、测试、样本与评估结果

当前测试覆盖已经包括：

- backend adapter 委托 engine client。
- task-center 和 task-engine 的状态回填。
- backend HTTP 集成路径。
- asset-scan bridge 与 runtime 执行上下文。
- FOFA task-scan、batch-report、naabu/nmap 主线脚本与样本导出。

阶段验证中已形成以下典型结果：

| 验证项 | 结果摘要 |
| --- | --- |
| 本地 Ollama live probe | 可命中 `127.0.0.1:11434`，结果包含开放端口、`/api/tags` endpoint 和 Ollama 指纹 |
| AutoGPT 反面测试 | 本地未运行服务时，fingerprint 为空，符合负向验证预期 |
| LangFlow live probe | 支持通过 URL 端口与 `probe_port_hint` 优先尝试实际运行端口 |
| Ollama round1 | `size=20`，创建任务 20 个，完成 20 个，失败 0 个，复核得到 8 条强正样本 |
| Ollama round3 | `size=50`，创建任务 50 个，完成 50 个，失败 0 个，复核得到 12 条强正样本，运输失败 22 条 |
| Ollama round5 至 round9 | high 组强正率较稳定，但 info 抽样运输失败占比偏高，暂不满足扩大到 `size=100` 的质量门槛 |
| naabu+nmap round1 | `size=50` 试跑中因 naabu 外部依赖失败未进入 nmap 阶段，已保留失败证据 |
| naabu+nmap round2 | 增加回退后，50 个候选中 nmap 尝试 46 个，说明流程从“全量阻断”推进到“可进入服务验证阶段” |

从评估角度看，当前 Ollama 主线已经能够稳定产生强正样本，但强负样本仍不足；info 组运输失败比例偏高，是后续继续扩容前最需要治理的问题。

## 七、当前阶段成果总结

本阶段已经形成以下可提交成果：

- 资产测绘结果结构已经从文档整理转化为工程类型和结果输出。
- `asset-scan` 规则已升级到 v2，并具备面向多类智能体服务的扩展能力。
- 引擎已具备六步最小闭环，不再只是单个探针或单个样例。
- 后端已通过统一任务 API 接入 `asset-scan`，保持平台入口一致。
- FOFA 候选已能进入 task-scan 和后端任务链路，形成公网候选到平台任务的转换路径。
- naabu、nmap、HTTP `/api/tags` 已形成分层验证与回退流程，并保留原始证据。
- 已建立强正样本、强负样本、运输失败三类评估口径，为后续规则回归和查询模板对比提供基础。

## 八、当前边界与问题

当前阶段仍有明确边界：

- 不做无授权的全网扩散扫描。
- 不新增与当前任务无关的公开接口。
- 不把 FOFA、naabu、nmap 的 dev 侧脚本包装成正式平台 API。
- 不把运输失败样本误判为负样本。
- 暂不推进分布式扫描调度和完整前端报表页面。

主要待解决问题包括：

- strong negative 样本数量不足，需要专项补采。
- info 组候选运输失败占比较高，需要做失败分桶分析和 query 收敛。
- naabu/nmap 依赖本地工具和网络环境，实际执行时需要继续完善超时、回退和审计策略。
- LangFlow、AutoGPT、OpenClaw Gateway 目前只保留旁线验证记录，暂未成为主线样本扩展对象。

## 九、下一阶段计划

后续建议优先按“质量优先、再扩规模”的原则推进：

1. 继续以 Ollama 为唯一主线，保持 `size=50` 的稳定基线。
2. 对 round5 至 round9 的运输失败做分桶分析，区分 timeout、fetch failed、TLS/refused 等原因。
3. 做 2 到 3 组 FOFA query A/B 对比，优先降低 info 组运输失败比例。
4. 专项补采 strong negative 样本，确保规则评估不只依赖正样本。
5. 冻结一版正样本、负样本和运输失败评测集，作为后续规则变更的回归基线。
6. 在连续两轮满足质量门槛后，再考虑从 `size=50` 放大到 `size=100` 或更高批次。

## 十、阶段结论

总体来看，本阶段已经完成了从“资产测绘思路”到“平台可执行闭环”的关键转化：上游有 FOFA 候选来源，中间有 task-scan 和后端任务中心承接，下游有 naabu/nmap/HTTP endpoint 的证据链，以及正负样本和评估指标作为反馈。当前成果已经具备继续扩展的工程基础，也保留了授权、预算、审计和样本准入等安全边界。

下一阶段不宜盲目扩大扫描规模，而应先治理查询质量和样本结构，使资产测绘链路从“能跑通”进一步走向“可稳定评估、可持续迭代”。
